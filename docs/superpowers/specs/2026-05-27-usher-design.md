# usher Design

## Summary

`usher` is a headless, self-hosted credential vault and authenticated HTTP caller for agents. It keeps OAuth refresh tokens, bearer tokens, and related secrets out of the agent runtime. Agents call external APIs through `usher`; `usher` resolves the right credential, injects authorization, forwards the request, and returns the upstream response as directly as possible.

Version 1 is single-user and provider/API agnostic. API contract awareness stays with the agent.

## Goals

- Store credential material securely outside the agent runtime.
- Support generic authenticated HTTP calls without provider-specific API tools.
- Let agents call APIs without knowing credential IDs.
- Support `OAuth2` and `BearerToken` credentials from the start.
- Keep the service headless and self-hostable.
- Use SQLite for persistence.
- Use strict key-file-based encryption for stored secrets.

## Non-Goals

- No MCP adapter in v1.
- No admin/user UI in v1.
- No multi-user SaaS model in v1.
- No app-layer authentication in v1.
- No credential updates in v1.
- No provider-specific API semantics in `usher`.
- No persisted access tokens.

## Architecture

`usher` has three main responsibilities:

- Store credentials securely in SQLite.
- Resolve incoming `/call` requests to exactly one active credential using URL matchers.
- Inject auth and forward requests so agents receive upstream API responses as directly as possible.

Credentials are the single top-level auth concept. There are no separate provider, connection, issuer, or OAuth app resources in v1.

## API Surfaces

Admin credential endpoints are loopback-only:

```http
GET /credentials
POST /credentials
GET /credentials/{credentialId}
DELETE /credentials/{credentialId}
```

OAuth browser endpoints may be externally reachable when needed:

```http
GET /credentials/{credentialId}/oauth2/login
GET /oauth2/callback
```

Authenticated external API calls use:

```http
<method> /call?url=<absolute-https-target-url>
```

`/call` is available to loopback plus configured allowed caller IPs.

## Credential Model

Credential IDs are always generated by `usher`. They must be opaque and high entropy because OAuth login URLs may be externally reachable. Agents do not pass credential IDs to `/call`.

Common credential fields:

```json
{
  "credentialId": "cred_...",
  "type": "OAuth2",
  "label": "Google Calendar",
  "status": "pending",
  "allowedRequests": [
    {
      "url": {
        "origin": "https://www.googleapis.com",
        "pathPrefix": "/calendar/"
      }
    }
  ],
  "createdAt": "2026-05-27T00:00:00Z",
  "updatedAt": "2026-05-27T00:00:00Z"
}
```

Statuses are single words:

```json
"pending"
"active"
"error"
```

Deletion is the only lifecycle operation beyond creation and OAuth activation. Changing label, scopes, client secret, token material, or allowed request matchers requires deleting and recreating the credential.

## Allowed Request Matching

Each credential has one or more structured URL matchers:

```json
{
  "url": {
    "origin": "https://www.googleapis.com",
    "pathPrefix": "/calendar/"
  }
}
```

V1 matching rules:

- `origin` matches exact scheme, host, and optional port.
- Only `https://` origins are accepted.
- `pathPrefix` must start and end with `/`.
- Query strings are ignored for credential matching.
- URL fragments are rejected for `/call` targets.
- Registration rejects overlapping matchers across all non-deleted credentials.
- Overlap exists when origins are equal and either path prefix starts with the other path prefix.

This guarantees `/call` can resolve at most one active credential for a target URL.

## OAuth2 Credentials

Creating an OAuth2 credential:

```http
POST /credentials
```

```json
{
  "type": "OAuth2",
  "label": "Google Calendar",
  "allowedRequests": [
    {
      "url": {
        "origin": "https://www.googleapis.com",
        "pathPrefix": "/calendar/"
      }
    }
  ],
  "oauth2": {
    "clientId": "...",
    "clientSecret": "...",
    "authorizationUrl": "https://accounts.google.com/o/oauth2/v2/auth",
    "tokenUrl": "https://oauth2.googleapis.com/token",
    "scopes": ["https://www.googleapis.com/auth/calendar.readonly"]
  }
}
```

The returned representation redacts secrets and includes a login URL:

```json
{
  "credentialId": "cred_...",
  "type": "OAuth2",
  "label": "Google Calendar",
  "status": "pending",
  "allowedRequests": [
    {
      "url": {
        "origin": "https://www.googleapis.com",
        "pathPrefix": "/calendar/"
      }
    }
  ],
  "oauth2": {
    "clientId": "...",
    "clientSecretPreview": "****abcd",
    "authorizationUrl": "https://accounts.google.com/o/oauth2/v2/auth",
    "tokenUrl": "https://oauth2.googleapis.com/token",
    "scopes": ["https://www.googleapis.com/auth/calendar.readonly"],
    "grantedScopes": []
  },
  "loginUrl": "https://usher.example.com/credentials/cred_.../oauth2/login"
}
```

OAuth login:

```http
GET /credentials/{credentialId}/oauth2/login
```

Behavior:

- Browser-oriented endpoint.
- Creates a short-lived, one-time OAuth state.
- Redirects to the configured provider authorization URL.
- Only works for `OAuth2` credentials with status `pending` or `error`.
- Relies on opaque high-entropy credential IDs plus short-lived OAuth state; credential creation remains loopback-only.

OAuth callback:

```http
GET /oauth2/callback
```

Behavior:

- Validates state.
- Exchanges authorization code for tokens.
- Stores refresh token encrypted.
- Stores granted scopes if returned by the provider.
- Marks credential `active` on success.
- Returns a minimal browser-readable success or failure page.
- Never exposes tokens.

During `/call`, OAuth2 credentials obtain or refresh access tokens as needed. Access tokens may be cached in memory until expiration but are not persisted.

## BearerToken Credentials

Creating a bearer token credential:

```http
POST /credentials
```

```json
{
  "type": "BearerToken",
  "label": "Internal API",
  "allowedRequests": [
    {
      "url": {
        "origin": "https://api.internal.example.com",
        "pathPrefix": "/"
      }
    }
  ],
  "bearerToken": {
    "token": "..."
  }
}
```

Returned representation redacts the token:

```json
{
  "credentialId": "cred_...",
  "type": "BearerToken",
  "label": "Internal API",
  "status": "active",
  "allowedRequests": [
    {
      "url": {
        "origin": "https://api.internal.example.com",
        "pathPrefix": "/"
      }
    }
  ],
  "bearerToken": {
    "tokenPreview": "****abcd"
  }
}
```

During `/call`, `usher` injects the stored token as:

```http
Authorization: Bearer <token>
```

## `/call` Flow

Example request:

```http
GET /call?url=https%3A%2F%2Fwww.googleapis.com%2Fcalendar%2Fv3%2Fusers%2Fme%2FcalendarList
User-Agent: my-agent/1.0
Accept: application/json
```

Validation before proxying:

- Caller IP must be loopback or match the configured `/call` allowlist.
- `url` query parameter is required.
- Target URL must be absolute `https://`.
- Target URL must not contain a fragment.
- `User-Agent` header is required and non-empty.
- Caller must not provide `Authorization`.
- Hop-by-hop headers are ignored or stripped.
- Exactly one active credential must match the target URL.

Forwarding behavior:

- Original HTTP method is used.
- Request body is forwarded as-is.
- Most request headers are forwarded as-is.
- `usher` injects `Authorization: Bearer ...`.

Response behavior:

- If upstream responds, return upstream status code, body, and headers as directly as possible.
- No JSON envelope for upstream responses.
- Strip hop-by-hop response headers.
- Do not expose credential material.

If `usher` itself fails, it returns an error with explicit headers:

```http
x-usher-error: true
x-usher-error-code: NoMatchingCredentialError
```

Error body:

```json
{
  "error": {
    "code": "NoMatchingCredentialError",
    "message": "No credential allows requests to this URL."
  }
}
```

Error codes are PascalCased and end with `Error`.

## Network Access

Admin endpoints are loopback-only:

- `127.0.0.1`
- `::1`

OAuth browser endpoints may be externally reachable because OAuth provider redirects may need to reach them through a public URL or temporary tunnel.

`/call` allows:

- loopback callers
- source IPs configured in the call allowlist

No app-layer authentication is included in v1.

## Storage And Encryption

SQLite stores:

- credential metadata
- allowed request matchers
- OAuth config
- encrypted secret blobs
- OAuth state records
- audit logs

Encryption key handling:

- `USHER_ENCRYPTION_KEY_FILE` is mandatory.
- Raw encryption keys in environment variables are not supported.
- No dev-mode fallback exists.
- The key file must be owned by the same effective user running `usher`.
- The key file must not grant group or other access.
- Acceptable file modes are `0400` and `0600`.
- The file contains exactly one line in the format `base64url:<value>`, where `<value>` decodes to exactly 32 bytes.
- The key is read once at startup and kept only in process memory.

Secret encryption:

- Use the key file value as a master key.
- Derive per-value keys with HKDF using credential ID, secret purpose, and encryption version.
- Encrypt using authenticated encryption.
- Use a fresh nonce per encrypted value.
- Use associated data so ciphertext cannot be moved between credentials or fields.
- Store encryption metadata with ciphertext.

Raw `clientSecret`, refresh tokens, bearer tokens, and access tokens are never returned by any API and are never logged.

## Audit Logs

`/call` attempts and outcomes are audit logged.

Audit records include:

- timestamp
- source IP
- `User-Agent`
- HTTP method
- target URL
- matched credential ID, when available
- upstream status or `usher` error code
- outcome

Sensitive headers are redacted. `Authorization` is never logged.

## Error Codes

Initial v1 semantic errors include:

- `CallerIpNotAllowedError`
- `MissingUrlError`
- `InvalidTargetUrlError`
- `MissingUserAgentError`
- `ReservedHeaderError`
- `NoMatchingCredentialError`
- `OverlappingAllowedRequestError`
- `CredentialNotFoundError`
- `InvalidCredentialTypeError`
- `InvalidCredentialStatusError`
- `OAuthStateInvalidError`
- `OAuthTokenExchangeFailedError`
- `EncryptionKeyFileMissingError`
- `EncryptionKeyFileNotOwnedByProcessUserError`
- `EncryptionKeyFileTooPermissiveError`
- `EncryptionKeyInvalidFormatError`

Errors are implemented as separate semantic error classes, not one generic error type. All `usher` errors include PascalCased error codes ending in `Error` in both response headers and body where a body is returned.

## Testing Expectations

V1 should include tests for:

- Credential matcher overlap detection.
- `/call` routing for no match and one match.
- Overlap rejection during credential creation.
- `Authorization` header rejection.
- Mandatory `User-Agent` validation.
- IP allowlist behavior.
- OAuth2 state validation and callback handling.
- Key file permission validation.
- Secret encryption and decryption without exposing raw secrets.
- `BearerToken` `/call` against a fake upstream server.
- `OAuth2` token refresh against a fake OAuth server.
